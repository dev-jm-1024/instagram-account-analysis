## Gemini

로컬 파일 시스템을 '네트워크 없는 DB'로 활용하여 프론트엔드 단독으로 구동되는 앱을 기획하신다면, 일반적인 웹 서비스와는 데이터의 흐름(Data Flow)과 **상태 관리(State Management)** 방식이 완전히 달라집니다.

서버(API)를 호출하는 대신, 디스크에서 파일을 읽어와 메모리에 올리고 화면을 전환하는 구조가 되어야 합니다. 이를 바탕으로 필수적인 페이지 구성과 각 화면 간의 데이터 입출력(I/O) 프로세스를 정리해 드립니다.

---

### 🗺️ 페이지 구성 및 데이터 I/O 설계

#### **1. 🏠 랜딩 및 데이터 로드 페이지 (데이터베이스 연결 역할)**

가장 먼저 사용자가 자신의 인스타그램 데이터를 앱에 제공(연결)하는 관문입니다.

* **Input (입력):** 사용자가 브라우저의 File System API(`showDirectoryPicker`)를 통해 압축을 푼 인스타그램 폴더를 선택합니다.
* **Process (처리):** * 선택된 폴더 내부 구조가 유효한지 검사합니다 (예: `messages`, `followers_and_following` 폴더 존재 여부 확인).
* 가장 가벼운 기본 프로필 정보(`personal_information.json`)를 읽어와 파싱합니다.


* **Output (출력 및 상태 저장):** * 사용자의 **디렉토리 핸들(Directory Handle)** 객체를 프론트엔드의 전역 상태(Global State)에 저장합니다. (이후 모든 페이지는 이 핸들을 통해 파일을 읽습니다.)
* "환영합니다, OOO님" 메시지와 함께 대시보드로 라우팅합니다.



#### **2. 📊 메인 대시보드 (요약 및 네비게이션)**

전체적인 내 데이터의 규모를 한눈에 보여주고, 각 상세 메뉴로 이동하는 허브입니다.

* **Input:** 전역 상태에 저장된 디렉토리 핸들.
* **Process:** 각 디렉토리의 파일들을 비동기(Async)로 훑어보며 데이터의 '크기'나 '개수'만 빠르게 카운팅합니다. (예: `followers.json`의 배열 길이 = 총 팔로워 수).
* **Output:** 팔로워 수, 총 DM 방 개수, 최근 로그인 시간 등 핵심 요약 지표를 카드 형태로 렌더링합니다.

#### **3. 👥 인적 네트워크 분석 페이지 (팔로우/팔로잉)**

* **Input:** `followers_and_following` 폴더 내의 `followers.json` 및 `following.json` 파일 읽기.
* **Process (데이터 가공):** * 읽어온 두 JSON 객체의 배열을 교차 검증하는 로직(알고리즘)이 실행됩니다.
* `팔로잉 집합 - 팔로워 집합 = 맞팔이 아닌 사람 (내가 일방적으로 팔로우함)`


* **Output:** 교차 검증이 완료된 리스트를 메모리에 올리고, 화면에는 '전체 보기', '나만 팔로우하는 계정', '날 팔로우하지 않는 계정' 탭으로 나누어 리스트 UI로 출력합니다.

#### **4. 💬 다이렉트 메시지 (DM) 뷰어 페이지**

이 앱에서 가장 복잡하고 I/O가 활발한 페이지입니다. 좌측엔 채팅방 목록, 우측엔 대화창이 있는 스플릿 뷰(Split View) 구조가 적합합니다.

* **Input 1 (사이드바):** `messages/inbox` 폴더를 읽어 하위 폴더(채팅방) 이름들의 목록을 가져옵니다.
* **Output 1:** 채팅방 리스트를 화면 좌측에 렌더링합니다.
* **Input 2 (채팅방 클릭 시 이벤트):** 사용자가 특정 채팅방을 클릭하면, 해당 폴더 내의 `message_1.json`을 읽어옵니다.
* **Process (미디어 처리 핵심):** * JSON 내부에 텍스트 메시지는 그대로 디코딩합니다.
* 사진/영상 경로(`photos/xxx.jpg`)가 있다면, 디렉토리 핸들을 이용해 해당 실제 미디어 파일을 읽어들여 브라우저 메모리상에 Blob URL(`blob:http://localhost/...`)로 변환합니다.


* **Output 2:** 텍스트 말풍선과 변환된 미디어(이미지/영상 태그)를 시간순으로 렌더링합니다.

#### **5. 🖼️ 활동 내역 및 미디어 갤러리 페이지**

* **Input:** `content/posts.json` 및 `likes/liked_posts.json`.
* **Process:** * 용량이 큰 미디어 파일들을 한 번에 모두 Blob URL로 변환하면 브라우저 메모리가 터질 수 있습니다.
* JSON에서 경로 목록만 먼저 파싱해 두고, 화면을 스크롤할 때마다(Lazy Loading / Intersection Observer) 화면에 보이는 이미지만 디스크에서 읽어와 렌더링하는 최적화 처리가 필요합니다.


* **Output:** 인스타그램 피드나 그리드 형태의 갤러리 UI를 출력합니다.

#### **6. ⚙️ 보안 및 시스템 로그 페이지**

* **Input:** `login_and_account_creation` 폴더의 JSON 파일들.
* **Process:** 유닉스 타임스탬프(1717478400 등)로 되어 있는 시간 데이터를 사용자가 보기 편한 포맷(YYYY-MM-DD HH:mm:ss)으로 변환하고, 최신순으로 정렬(Sort)합니다.
* **Output:** 데이터 그리드(Table) 형태로 접속 IP, 기기 정보, 시간 등을 깔끔하게 나열합니다.

### 💡 아키텍처 설계 시 핵심 포인트 (프론트엔드 관점)

기존에 서버와 통신하는 프론트엔드 개발(React 등)에 익숙하시다면, 백엔드의 역할을 프론트엔드의 '서비스 레이어(Service Layer) 함수'들이 대신한다고 생각하시면 이해가 쉽습니다.

1. **Repository (데이터 접근):** File API를 이용해 JSON 파일을 텍스트나 스트림으로 읽어오는 함수들.
2. **Service (비즈니스 로직):** 깨진 유니코드를 복구하고, '맞팔 계산'이나 '날짜 포맷팅'을 수행하며 프론트엔드용 DTO 모델로 변환하는 함수들.
3. **View (렌더링):** 가공된 DTO 상태(State)를 기반으로 화면을 그리는 컴포넌트들.

이런 식으로 프론트엔드 내부에서 계층을 철저히 분리해두면, 나중에 파일 파싱 로직이나 데이터 구조가 바뀌더라도 화면을 그리는 UI 코드는 수정하지 않아도 되어 유지보수가 훨씬 수월해집니다.

## Claude

좋아요, 화면(페이지) 단위로 전체 흐름과 각 화면의 데이터 입출력(I/O)을 정리하겠습니다. 앞서 정한 7개 메뉴 + 진입/적재 화면까지 포함한 전체 플로우입니다.

## 전체 화면 플로우

```
[1.폴더 선택] → [2.임포트 진행] → [3.Overview] ──┬─→ [4.팔로우]
                                                    ├─→ [5.DM 통계]
                                                    ├─→ [6.활동 기록]
                                                    ├─→ [7.히트맵]
                                                    ├─→ [8.로그인 기록]
                                                    ├─→ [9.각종 로그]
                                                    └─→ [10.데이터 탐색기]
```

핵심 구조: **1·2는 1회성(ETL)**, 3~10은 **이미 DB에 적재된 데이터를 조회**만 함.

---

## 화면별 데이터 입출력

**[1] 폴더 선택 (랜딩)**
- 입력(사용자): 압축 푼 폴더 경로
- 출력(→백엔드): `POST /api/sessions { folderPath }`
- 응답(←백엔드): `{ sessionId, detectedMenus[], 추정 데이터량 }` — 폴더를 가볍게 스캔해서 "이 export엔 팔로우·DM·로그인 데이터가 있음"을 먼저 알려줌 (파일 내용은 안 읽음, 트리만)
- 화면 표시: 감지된 메뉴 체크리스트 + "분석 시작" 버튼
- 에러 처리: HTML export 감지 시 "JSON으로 다시 받으세요" 안내

**[2] 임포트 진행 (ETL)**
- 입력: sessionId, "시작" 트리거
- 동작: 백엔드가 스트리밍 파싱 → 정규화 → 임베디드 DB 적재
- 출력(←백엔드, 실시간): 진행률 (`SSE` 또는 폴링) — "팔로워 12,340건 적재… DM 1,203방 처리중…"
- 완료 시: `{ status: done, summary: {...} }` → Overview로 자동 이동
- 포인트: 여기가 유일하게 느린 구간(2GB). 이후 화면은 전부 DB라 빠름

**[3] Overview (요약 대시보드) — 진입 허브**
- 입력: `GET /api/{sid}/overview`
- 출력(←): 숫자 카드용 집계
    - 총 팔로워/팔로잉, 맞팔률
    - 총 게시물/DM방/좋아요/댓글 수
    - 계정 사용기간(최초~최근), 가장 활발한 달
- 화면→화면: 각 카드 클릭 → 해당 상세 메뉴로 이동 (drill-down)

**[4] 팔로우 보기**
- 입력: `GET /api/{sid}/connections?filter=not_following_back&sort=timestamp&page=0`
- 출력(←): `{ stats:{팔로워수,팔로잉수,맞팔,안맞팔}, list:[{username, followedAt, isMutual}], pageInfo }`
- 화면 입력(사용자): 필터(안맞팔/맞팔/내가짝사랑/최근언팔), 정렬, 검색, 페이지
- 출력(→): 필터·정렬·페이지 파라미터 → 재조회 (DB 쿼리라 즉시)

**[5] DM 통계**
- 입력: `GET /api/{sid}/messages/summary`, 상세는 `GET /api/{sid}/messages/{roomId}`
- 출력(←): `{ totalRooms, topPartners:[{name, msgCount, sentRatio, initiatorRatio}], hourlyDist[] }`
- 화면→화면: 상대 클릭 → 그 방 상세(시간추이, 보낸/받은 비율) drill-down
- 사용자 입력: Top N 개수, 기간 필터

**[6] 활동 기록 (게시물·좋아요·댓글)**
- 입력: `GET /api/{sid}/activity?type=posts|likes|comments&from=&to=`
- 출력(←): `{ monthlyCounts:[{month, posts, likes, comments}], totals:{} }`
- 사용자 입력: 활동 타입 토글, 기간 범위
- 화면: 월별 스택 막대그래프 + 타입별 토글

**[7] 활동 히트맵**
- 입력: `GET /api/{sid}/heatmap?sources=posts,likes,dm,login`
- 출력(←): `{ grid:[7][24] 카운트, peak:{day,hour} }`
- 사용자 입력: 어떤 활동 종류를 히트맵에 포함할지 체크박스
- 화면: 요일×시간 7×24 히트맵, "가장 활발한 시간" 하이라이트

**[8] 로그인 기록**
- 입력: `GET /api/{sid}/logins`
- 출력(←): `{ timeline:[{time, ip, device}], byDevice:[], byIp:[] }`
- 화면: 시간순 타임라인 + 기기별/IP별 집계. (가능하면 IP→지역 매핑은 오프라인 GeoIP DB 필요 — 네트워크 안 쓰려면 별도 고려)

**[9] 각종 로그**
- 입력: `GET /api/{sid}/logs?file={경로}`
- 출력(←): 범용 키-값 테이블 + timestamp 정렬된 행들
- 화면: 비표준 데이터라 **범용 테이블 뷰어** (정렬·검색·페이징만)
- 사용자 입력: 로그 파일 선택, 컬럼 정렬/검색

**[10] 데이터 탐색기 (Raw Viewer)**
- 입력: `GET /api/{sid}/tree` (디렉토리 트리), `GET /api/{sid}/raw?path=...` (파일 내용)
- 출력(←): 트리 구조 / 선택 파일의 원본 JSON + 추론된 스키마(키·타입·샘플값)
- 화면: 좌측 트리, 우측 JSON 뷰어 + 스키마 인스펙터
- 역할: 메인 메뉴가 못 잡은 파일 fallback, 디버깅

## 공통 I/O 규칙

- **세션 기반**: 1번에서 받은 `sessionId`가 이후 모든 요청에 따라붙음 (어느 DB/폴더를 보는지 식별)
- **상태 공유**: sessionId, 감지된 메뉴 목록, summary는 프론트 전역상태(Context/Zustand 등)에 보관 → 화면 전환마다 재요청 안 함
- **드릴다운 패턴**: Overview 카드 → 상세 메뉴, DM 요약 → 개별 대화방. 큰 그림에서 세부로 파고드는 일관된 네비게이션
- **페이징**: 팔로우·DM·로그처럼 목록 큰 건 전부 서버 페이징 (한 번에 다 안 내림)

## Claud Code

`domain.md`(스펙 원본)를 재정독하며 두 의견을 대조 검토했다. 좋은 통찰이 많지만 **각각 스펙과 정면 충돌하는 가정**이 있어, ① 두 의견 교정 → ② 정정판 화면 지도 → ③ 정밀 I/O 의존 순으로 정리한다.

### 1. 앱의 본질 — I/O 설계를 지배하는 전제

| 전제 | I/O 에 미치는 영향 |
|------|------------------|
| Spring Boot 단일 JAR + React(static 동봉) | 프론트는 **REST 호출만**, 디스크 접근은 백엔드 전담 |
| 사용자 1명·localhost | **세션/인증/sessionId 없음** — 모든 요청이 같은 인메모리 보관소를 봄 |
| **DB 없음** | 최초 1회 파싱 → `@Service` 빈 메모리 적재 → Java Stream 집계 |
| **전 데이터 읽기 전용** | 유일한 쓰기 = 임포트. 모든 메뉴는 조회(GET)뿐 |
| 미디어 바이너리·DM 원문 비노출 | 갤러리/채팅뷰어 화면 **없음**. 메타데이터·통계만 |

### 2. 두 의견의 스펙 위반 지점 (반드시 교정)

**Gemini — 아키텍처 전제가 통째로 틀림**

- "프론트엔드 단독 + 브라우저 File System API(`showDirectoryPicker`, Directory Handle, Blob URL)" 구조로 봤다. 그러나 스펙은 **Spring Boot 백엔드 + 인메모리 `@Service` + REST API**다. 브라우저가 디스크를 직접 읽지 않는다 → Gemini의 I/O 모델 전체가 부적합.
- **DM 대화 뷰어 + 미디어 갤러리**는 스펙 직접 위반: DM은 *"원본 텍스트 비노출·통계만, content 비보관"*, 미디어는 *"media/ 바이너리 파싱 제외"*. 채팅창·Blob 변환·갤러리는 대상 밖.
- 단 **계층 분리(Repository/Service/View)** 조언은 옳다 — 그 계층을 프론트가 아닌 **백엔드(Spring)** 에 둔다.

**Claude — 방향은 맞으나 4가지 어긋남**

- `sessionId`/멀티세션 → 스펙은 *"세션·멀티유저 없음"*. 부록 A 엔드포인트도 전부 평면 경로(`/api/follows`).
- "임베디드 DB / DB 쿼리" → 스펙은 *"DB 없음 — 인메모리 + Java Stream"*.
- **메뉴 7개로 셈** → 실제 **9개**. **검색 기록(Search)** 누락.
- 로그인 **GeoIP(IP→지역)** 제안 → 스펙은 *"외부 네트워크 없음 + 지역 변환 안 함"*.
- 단 **드릴다운·전역 상태 캐싱** 패턴은 적절(페이징은 스펙 외이나 대량 목록엔 권장).

### 3. 정정판 화면 지도 — **임포트 + 본인식별 + 9개 메뉴 = 11화면**

```
■ 진입 게이트
  임포트(폴더경로 입력) ──IN_PROGRESS중 메뉴잠금──┐
        │ COMPLETED                               │
        │ + ownerResolved:false → 본인 username 수동입력
        ▼
[0.개요 대시보드] ◀── 진입 첫 화면 / 카드 클릭 시 drill-down
   ├─ 1.팔로우 관계   ├─ 4.히트맵        ├─ 7.각종 로그
   ├─ 2.DM 통계       ├─ 5.검색 기록     └─ 8.데이터 탐색기
   ├─ 3.활동 기록     ├─ 6.로그인 기록
       (게시물/좋아요/댓글/저장 4 서브탭)
```

> `personal_information/` 는 메뉴 아님 — 본인 식별 목적으로만 최소 파싱.

**상태머신**

```
        POST /api/import
IDLE ──────────────▶ IN_PROGRESS ──파싱──▶ COMPLETED ──┬─ ownerResolved:true  → 9메뉴 전부 활성
  ▲ 재임포트(덮어쓰기)     │ (메뉴 잠금)                 └─ ownerResolved:false → 메뉴 활성 + DM만 수동입력 요구
  └────────────────────── │ ──치명오류──▶ FAILED
```

### 4. 화면별 I/O (부록 A 기준, 세션 없음)

| 화면 | IN | OUT | 데이터 출처 |
|------|----|-----|------------|
| 임포트 | 폴더경로 `POST /api/import` | 진행률·상태값·소요시간·파싱수·경고·`ownerResolved` | 직접 파싱(ETL) |
| 본인식별 | username `POST /api/import/owner` | `ownerResolved:true` | 수동 fallback |
| 개요 | `GET /api/overview` | 팔로워/팔로잉/맞팔/짝사랑·총게시물(post+story+reels)·DM방+메시지·좋아요+댓글·**활동기간**·**가장 활발한 달** | 타 도메인 카운트 재활용 |
| 팔로우 | `GET /api/follows?type=mutual\|i_follow\|follows_me\|unfollowed\|...` | type별 username+팔로우시각, 맞팔(∩)/짝사랑(차집합) | 자체 파싱 |
| DM | `GET /api/messages/stats` | 대화방수·상대별 메시지수·보낸/받은 비율·Top10·initiator·시간대(0\~23) | 자체 파싱 + **본인 displayName 의존** |
| 활동 | `GET /api/activity?type=post\|like\|comment\|saved` | 날짜별 타임라인·월별 카운트(메타데이터만) | 자체 파싱 |
| 히트맵 | `GET /api/heatmap` | `int[7][24]`(KST)+peak | **게시물·좋아요·댓글·DM·로그인** ts 재활용 |
| 검색 | `GET /api/searches` | 검색어 빈도 내림차순 | 자체 파싱 |
| 로그인 | `GET /api/logins` | 최신순 timestamp·ip·user_agent (지역변환 X, LOGIN/LOGOUT) | 자체 파싱 |
| 각종 로그 | `GET /api/logs` | 파일별 그룹 키-값 테이블·ts 정렬 | 검색이 안 가져간 나머지 |
| 탐색기 | `GET /api/explorer/tree` · `/file?path=` | 트리(미디어 제외·깊이10) / 원본 JSON(구조 raw·문자열만 보정·10MB truncate·루트밖 400·미존재 404) | 원본 트리 fallback |

### 5. 화면 간 데이터 의존 — 설계의 뼈대 4가지

1. **임포트 → 전 화면 게이트**: `COMPLETED` 아니면 프론트가 메뉴 잠금. 미완료 조회 분기가 도메인마다 다름 — 개요 `importRequired:true`(200) / 히트맵 `503`.
2. **본인 식별 2형태 → 도메인별 매칭키가 다름**:
   - `username` ↔ 팔로우의 `value` 매칭
   - `displayName` ↔ DM의 `sender_name` 매칭
   → 두 형태를 **모두** 보관. `ownerResolved:false` 영향은 **DM 한정**(보낸/받은·initiator 보류), 다른 메뉴는 무관. (Gemini·Claude 둘 다 이 fallback 화면을 누락)
3. **재활용 도메인은 입력 소스가 서로 다름** (← 흔히 뭉뚱그리는 지점):
   - **히트맵** = 게시물·좋아요·댓글·DM·로그인 **5종** (검색·저장 제외)
   - **개요** = 팔로우·활동·DM·좋아요·댓글 카운트 (로그인·검색은 카드에 없음)
   - **탐색기** = 원본 파일 트리 (집계 아닌 raw)
4. **claim 충돌 순서**: 검색(5)이 `logged_information/` 검색 파일을 **먼저** claim → 각종 로그(7)는 나머지만 (이중 집계 방지).

### 6. 타임스탬프 단위 — I/O 정합성의 숨은 함정

| 출처 | 원본 단위 | 정규화 |
|------|----------|--------|
| connections/ (팔로우) | **초** | ×1000 |
| messages/ (DM) | **ms** | 그대로 |
| 게시물·좋아요·댓글·저장·로그인 | **초** | ×1000 |

기준 `< 10_000_000_000L` → 초로 간주. **전 도메인 epoch ms(Long) 통일**은 임포트 화면(1.3)에서 끝내고, 이후 화면은 정규화된 값만 본다. 단위가 혼재하면 히트맵·개요 합산이 오염됨.

### 공통 I/O 규칙 (스펙 정합)

- **세션 ID 없음** — 모든 요청이 같은 인메모리 보관소를 본다. (Claude의 `sessionId` 제거)
- **상태 공유** — 임포트 상태·`ownerResolved`·개요 summary는 프론트 전역 상태에 캐싱, 메뉴 전환마다 재요청 안 함. (Claude 채택)
- **드릴다운** — 개요 카드 → 상세 메뉴. 단 DM은 "방별 통계"까지만, **원문 대화 열람 없음**.
- **대량 목록(팔로워·좋아요)** — 서버 페이징/가상 스크롤 권장(스펙 외, 실무상 필요).
- **계층 분리는 백엔드에** — Repository(스캔·스트리밍 파싱) → Service(정규화·집계·DTO) → Controller(REST). (Gemini 조언을 백엔드로 이식)

**한 줄 요약**: Gemini는 *아키텍처(프론트 단독)·프라이버시 범위(DM·미디어)* 를 반대로 잡았고, Claude는 *세션·DB·검색메뉴·GeoIP* 에서 어긋났다. 정정판 핵심은 **백엔드 인메모리 + REST + 9메뉴 + 통계만(원문/미디어 비노출) + 본인식별 fallback**, 그리고 **히트맵 5종 소스·개요 카드 범위·본인식별 2형태·타임스탬프 단위 통일**이다.
