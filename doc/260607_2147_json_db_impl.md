## 0. 설계 원칙 (포트 & 어댑터 / 헥사고날)

- **application = 포트(인터페이스)만 의존** (DIP). 도메인 집계 "의도"를 메서드로 노출.
- **infrastructure = 어댑터(구현체)**. DuckDB SQL/Appender 가 여기 갇힌다.
- **도메인별 포트로 분리** (ISP) — 서비스는 자기 포트만 본다.
- **읽기/쓰기/상태를 분리** (SRP) — Query 포트 ↔ Write 포트 ↔ State 스토어.
- **LSP** — 같은 포트의 file-DuckDB ↔ in-memory-DuckDB(테스트) 구현 교체 가능.

```
api (Controller/Assembler/DTO)
   │  depends on
application (Service)  ──uses──▶  PORTS (interface)
                                   ├ 읽기: *QueryPort (도메인별)
                                   ├ 쓰기: ImportWritePort
                                   ├ 상태: ImportStateStore
                                   └ 적재: DomainSink<T> (Phase 2)
                                        ▲ implemented by
infrastructure/persistence/duckdb (ADAPTERS)
   DuckDbConfig · ConnectionManager · SchemaManager · BulkWriter
   · Duck*Repository · DuckDbImportWriteAdapter · StartupStateLoader
domain (불변 모델/VO + 신규 Page<T>, ImportMetaKey)
```

## 2. application 레이어 — 포트(인터페이스) 정의와 역할

### 2.1 상태 포트 — `application/store/ImportStateStore`
**역할**: 임포트 진행 상태·소유자·스캔결과 등 **작고 자주 바뀌는 메타**를 보관(인메모리). 데이터(행) 책임 없음.
```java
public interface ImportStateStore {
    ImportResult importResult();
    boolean isCompleted();
    boolean isOwnerResolved();
    Optional<AccountIdentity> owner();
    Optional<Path> importRoot();
    boolean sourceExists(DomainType domain);
    List<Path> scannedFiles(DomainType domain);
    // 상태 전이(발행 변수는 항상 마지막) — 기존 InMemoryImportStore 의 status 부분 이관
    void markInProgress(Path importRoot, Map<DomainType,List<Path>> scanned);
    void publishCompleted(ImportResult completed, Map<DomainType,List<Path>> scanned);
    void markFailed(List<ParseWarning> warnings);
    void applyOwner(AccountIdentity owner);
    void reset();
}
```
**구현**: `InMemoryImportStateStore` — 기존 `InMemoryImportStore` 의 `volatile` 발행 순서(데이터→발행변수 마지막) 패턴을 상태에 한정해 유지.

### 2.2 읽기 포트(도메인별) — `application/query/*`
각 메서드는 **도메인 개념 1:1**(원시 행 덤프 아님). 행-목록은 `Page<T>` 로 페이지네이션.

| 포트 | 메서드 | 역할(→ SQL 개념) |
|---|---|---|
| `ActivityQueryPort` | `long count(ActivityType)` · `Map<YearMonth,Long> monthlyCounts(ActivityType)` · `Page<Post> posts(int,int)` · `Page<LikeEntry> likes(int,int)` · `Page<CommentEntry> comments(int,int)` · `Page<SavedPost> savedPosts(int,int)` | 타입별 카운트·월별 GROUP BY·행 페이지드 조회 |
| `MessageQueryPort` | `MessageStats stats(int topN)` | rooms count·sum(total)·top-N·시간대 GROUP BY |
| `FollowQueryPort` | `FollowAnalysis analyze()`(카운트만) · `Page<FollowEntry> findByType(FollowQueryType,int,int)` · `long count(FollowRelationType)` | username SQL 집합연산(mutual/iFollowOnly/followsMeOnly) + 타입필터 |
| `HeatmapQueryPort` | `ActivityHeatmap heatmap()` | heatmap_cell SELECT + peak |
| `LoginQueryPort` | `Page<LoginEvent> timeline(int,int)` · `long count()` | ts DESC 페이지드 |
| `SearchQueryPort` | `Page<SearchFrequency> frequencies(int,int)` · `long distinctCount()` | keyword GROUP BY count DESC |
| `MiscLogQueryPort` | `Page<LogRecord> records(int,int)` · `List<String> files()` | 로그 행 페이지드 |
| `OverviewQueryPort` | `long[] activityTotals()` · `long[] activityRange()`(min/max ts) | 요약 카드용 합계·전체 활동기간 |

> **결정 반영**: `FollowAnalysis`는 리스트가 아닌 **카운트 보유**로 축소(대용량 set-diff 를 메모리에 올리지 않기 위해). 실제 항목은 `findByType` 페이지드로.

### 2.3 쓰기 포트 — `application/store/ImportWritePort` (유지, 역할 재정의)
**역할**: ETL 산출물을 **원자적으로** 영속. 구현이 트랜잭션·벌크삽입·상태발행을 조율.
```java
public interface ImportWritePort {
    void markInProgress(Path importRoot, Map<DomainType,List<Path>> scanned);
    void replaceAll(ImportSnapshot snapshot);          // tx: truncate→bulk insert→meta→commit→상태 COMPLETED 발행
    void applyOwner(AccountIdentity owner, List<Conversation> rebuilt); // conversation UPDATE + meta
    void markFailed(List<ParseWarning> warnings);
    void reset();                                      // truncate + 상태 IDLE
}
```

### 2.4 적재 싱크 — `application/ingest/DomainSink<T>` (Phase 2)
**역할**: 파서가 List 누적 대신 원소를 **흘려보내** 배치 적재(임포트 RAM 천장 제거).
```java
public interface DomainSink<T> extends AutoCloseable {
    void accept(T element);   // 내부 배치 버퍼 → 임계치 도달 시 Appender flush
    @Override void close();   // 잔여 flush
}
```

## 3. domain 레이어 — 신규 모델

- **`domain/common/Page<T>`** (VO): `List<T> items, long total, int limit, int offset`. 모든 행-목록 응답 표준.
- **`domain/common/ImportMetaKey`** (enum): `OWNER_USERNAME, OWNER_DISPLAY, OWNER_RESOLVED, COMPLETED_AT, DURATION_MS, PARSED_COUNT, IMPORT_ROOT, SCHEMA_VERSION, STATUS`. `import_meta` 키 오타 방지·중앙화.
- **`FollowAnalysis` 수정**: 리스트 필드 → 카운트 필드(mutualCount/iFollowOnlyCount/followsMeOnlyCount + 관계별 카운트).

## 4. infrastructure/persistence/duckdb — 어댑터(구현체)와 역할

| 클래스 | 책임(SRP) |
|---|---|
| **`DuckDbConfig`** | `@Configuration`. `InstagramProperties` 에서 DB 경로(`<dataRoot>/.analyze/index.duckdb`) 도출. ConnectionManager·JdbcClient·각 Repository 빈 구성. PRAGMA `memory_limit`·`threads`·세션 `TimeZone=시스템존` 설정. |
| **`DuckDbConnectionManager`** | **단일 공유 연결 보유 + 접근 직렬화**(R1). `JdbcClient` 제공, `createAppender(table)` 패스스루, 트랜잭션 begin/commit/rollback 경계 제공. (조회 빈도 낮은 단일 사용자라 직렬화로 충분; 후일 읽기 연결 확장 여지) |
| **`DuckDbSchemaManager`** | DDL 소유: `ensureSchema()`(테이블 생성/검증), `dropAll()`/`truncateAll()`, `currentSchemaVersion()` vs `import_meta`. 스키마는 코드 상수 버전과 1:1. |
| **`DuckDbBulkWriter`** | 테이블별 벌크삽입(Appender 우선, 실패 시 배치 PreparedStatement 폴백). `append(Post)`/`append(LikeEntry)`… + `flush()`. Phase2 `DomainSink` 가 이를 감싼다. |
| **`DuckSql`** | SQL 문자열 상수 + **타임존 헬퍼**(`epoch_ms(ts)` 를 시스템존으로 변환하는 표현식)로 R3(월/시 의미) 일원화. |
| **`DuckActivityRepository`** | `ActivityQueryPort` 구현. monthlyCounts=시스템존 `strftime(...,'%Y-%m')` GROUP BY 등. |
| **`DuckMessageRepository`** | `MessageQueryPort` 구현. stats=count·sum·top-N·`conversation_hour` GROUP BY hour. |
| **`DuckFollowRepository`** | `FollowQueryPort` 구현. analyze=FOLLOWING/FOLLOWER username 집합연산 카운트. findByType=관계필터/집합결과 페이지드. |
| **`DuckHeatmapRepository`** | `HeatmapQueryPort` 구현. heatmap_cell→7×24 grid + peak. |
| **`DuckLoginRepository`** | `LoginQueryPort` 구현. `ORDER BY ts DESC LIMIT/OFFSET`. |
| **`DuckSearchRepository`** | `SearchQueryPort` 구현. keyword GROUP BY count DESC. |
| **`DuckMiscLogRepository`** | `MiscLogQueryPort` 구현. (Phase3 generic_record/JSON). |
| **`DuckOverviewRepository`** | `OverviewQueryPort` 구현. 합계·`min/max ts` UNION. |
| **`DuckDbImportWriteAdapter`** | `ImportWritePort` 구현. ConnectionManager 트랜잭션 + BulkWriter + SchemaManager + **ImportStateStore**(상태 발행) 조율. replaceAll=truncate→bulk→meta→commit→`publishCompleted`. applyOwner=conversation UPDATE. |
| **`StartupStateLoader`** | `ApplicationRunner`. 부팅 시 `ensureSchema`; DB가 COMPLETED & schema_version 일치 & **import_root 존재**면 → 스캐너만 재실행해 `scannedFiles` 재구성(R4) 후 상태 COMPLETED 복원. 아니면 IDLE/재임포트. |

## 5. api 레이어 — 변경

- 행-데이터 엔드포인트에 `@RequestParam(defaultValue) int limit/offset` 추가: `FollowController`·`SearchController`·`LoginController`·`MiscLogController`. 집계 엔드포인트(activity/messages/heatmap/overview)는 무변경.
- 응답 DTO에 페이지 메타 포함: `PageResponse<T>{items,total,limit,offset}` 신규 또는 기존 응답 확장. Assembler 가 `Page<T>`→DTO 변환.

## 7. 빈 의존 그래프 (와이어링)

```
*ServiceImpl ── (주입) ──▶ *QueryPort (구현: Duck*Repository)
                          ▶ ImportStateStore (구현: InMemoryImportStateStore)
RealImportService ─▶ ImportWritePort (구현: DuckDbImportWriteAdapter)
                   ─▶ ImportStateStore  (status 조회)
DuckDbImportWriteAdapter ─▶ ConnectionManager, BulkWriter, SchemaManager, ImportStateStore
Duck*Repository          ─▶ ConnectionManager(JdbcClient), DuckSql
StartupStateLoader       ─▶ SchemaManager, ConnectionManager, ImportStateStore, FileScanner

```
> `RealImportService`·`ImportPipeline`·파서·스캐너·`ImportGuard` 는 포트에만 의존하므로 **시그니처 거의 무변경**(Phase1). Phase2서 파서가 `DomainSink` 를 받도록만 확장.

## 8. 단계별 구현 매핑

- **Phase 0**: DuckDbConfig·ConnectionManager·SchemaManager·DuckSql. R1(연결)·R2(DB위치)·R3(타임존)·R6(네이티브) 검증.
- **Phase 1a (수직 슬라이스=Activity)**: Page·ImportMetaKey, DuckBulkWriter(activity 테이블), DuckActivityRepository, DuckDbImportWriteAdapter(부분), InMemoryImportStateStore, StartupStateLoader, ActivityServiceImpl 전환. 골든값/메모리/재시작 검증 → off-ramp.
- **Phase 1b (확산)**: 나머지 7개 Repository + WriteAdapter 완성 + 서비스 전환 + 컨트롤러 페이징. `ImportReadStore`/`InMemoryImportStore` 삭제.
- **Phase 2**: DomainSink + 파서 sink 화 + ImportPipeline 직접 ingest + ImportSnapshot 축소.
- **Phase 3(선택)**: GenericRecordQueryPort + generic_record + 임포트 리포트.

## 9. SOLID 근거 (클래스 단위)

- **SRP**: 연결(ConnectionManager) / DDL(SchemaManager) / 쓰기(BulkWriter·WriteAdapter) / 읽기(Repository) / 상태(StateStore) / 복원(StartupLoader) 가 각각 한 가지 변경 이유만 가짐.
- **OCP**: 새 도메인 = QueryPort+Repository+테이블+(sink) 신규 작성, 기존 어댑터/서비스 무수정. 미지 파일은 YAML+generic_record 로 흡수.
- **LSP**: 모든 포트가 in-memory-DuckDB 구현으로도 동작(테스트). Repository 교체가 서비스에 투명.
- **ISP**: 서비스가 12+ 메서드 단일 스토어 대신 자기 도메인 포트(2~6 메서드)만 의존.
- **DIP**: application→포트 의존, infra가 구현 제공, DuckDbConfig가 조립. SQL은 infra에 완전 격리.

## 10. 위험 연계 (plan 파일 R1~R8 대응)

- R1 연결모델 → `DuckDbConnectionManager`(단일 직렬, Phase0 스파이크).
- R2 DB위치 → `DuckDbConfig` 가 `.analyze/`(discovery·retention 비대상) 고정.
- R3 타임존 → `DuckSql` 시스템존 표현 + 골든값.
- R4 재시작 소유자폴백 → `StartupStateLoader` 스캐너 재실행.
- R6 네이티브 → duckdb_jdbc 가 **win-x64 / mac-arm64 / mac-x64**(배포 JDK 매트릭스) + linux(docker) 네이티브 포함 확인.
- R7 테스트 → 서비스=fake QueryPort, repository=in-memory DuckDB.
```
