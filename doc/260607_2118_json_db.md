# Instagram Analyzer — 대용량 JSON 저장/조회 전략 · DB 도입 트레이드오프

> 작성: 2026-06-07 21:18 · 대상: 이 앱의 데이터 저장 계층을 설계/결정하려는 사람(개발자 본인)
> 한 줄 요약: **데이터 양도 스키마도 사용자마다 예측 불가**하다는 문제의 해법은 "전송 최적화"가 아니라
> **인메모리 객체 적재 → 임베디드 컬럼형 분석 스토어(DuckDB)로의 전환**이다. NoSQL은 portable 제약을
> 깨는 쪽으로 비용이 들고, 정작 끌리는 "유연한 스키마"는 DuckDB로도 얻는다.

---

## 0. 결론 먼저 (TL;DR)

- **진짜 병목은 "데이터 전송"이 아니라 "메모리"다.** 대시보드 응답은 이미 집계값(월별 카운트·Top10·히트맵)이라 작고, 앱은 localhost 단일 사용자라 전송은 사실상 공짜다. 한계는 `InMemoryImportStore`가 **모든 항목을 Java 객체 List로 RAM에 들고 있는 것**.
- **"사용자마다 양을 모른다"는 불확실성은 DuckDB를 골라야 하는 이유 그 자체다.** 양을 예측하는 대신, 비용을 데이터에 비례시키고 메모리에 상한을 두면(스트리밍 임포트 + 디스크 기반 컬럼형 스토어) 라이트 유저든 인플루언서든 동일하게 안전하다.
- **"내가 본 적 없는 JSON 파일" 문제는 2-tier로 푼다.** 아는 도메인은 큐레이션(타입드 테이블+전용 대시보드), 모르는 파일은 제네릭(범용 records 테이블)+raw로 우아하게 격하시키고, **임포트 리포트로 가시화**한다. 코드에 이미 토대가 있다(YAML 매핑 외부화 · MISC_LOG catch-all · Explorer raw · ParseWarnings).
- **DuckDB ↔ Spring Boot 연동은 좋다.** 공식 JDBC 드라이버 → `JdbcClient`로 붙인다. JPA/Hibernate로는 붙이지 않는다(공식 dialect 없음, 애초에 OLAP엔 ORM 부적합). 이 프로젝트는 **기존 JPA가 없는 그린필드 + 포트 추상화(`ImportReadStore`/`ImportWritePort`)** 라 교과서적으로 깔끔하게 들어간다.
- **NoSQL은 권하지 않는다.** 서버형(Mongo/ES)은 portable·무설치 모델을 깨고(비용이 엉뚱한 곳에 듦), 임베디드형(Nitrite/RocksDB)은 분석 집계가 약해 요청당 O(n) 문제로 회귀한다. 유연한 스키마는 DuckDB의 JSON 컬럼 + `read_json_auto`로 대체된다.

## 1. 문제 정의

현재 실측 데이터: **총 3.9GB, 가장 큰 단일 JSON 약 8MB / 12만 줄**(`stories_viewed.json` 등).
하지만 사용자에 따라:

- **인플루언서·헤비 유저**: 위보다 수십 배 더 많을 수 있음 (수천만~억 줄 시나리오).
- **라이트 유저**: 훨씬 적음.
- **미지의 파일**: 로케일·계정유형(비즈니스/개인)·export 시점·사용한 기능에 따라 **개발자가 가진 샘플에 없는 JSON 파일**을 사용자가 가질 수 있음.

→ 두 개의 독립된 문제다:
- **(A) 데이터 양을 예측할 수 없음.**
- **(B) 스키마/파일 종류를 전부 알 수 없음(열린 집합, open set).**

## 2. 현재 아키텍처 점검 (2026-06-07 코드 대조)

| 구간 | 현황 | 위치 | 평가 |
|---|---|---|---|
| 파싱(import) | Jackson 3 스트리밍, 원소 단위 순회 | `infrastructure/parse/AbstractJsonParser` | ✅ 파일 통째로 안 올림 |
| 저장 | **DB 없음.** 모든 항목을 Java 객체 List로 RAM 보관 | `application/store/InMemoryImportStore` | ⚠️ **확장의 벽** |
| 대시보드 API | 집계값만 반환 (월별 카운트/Top10/시간대/히트맵) | `ActivityResponse`, `MessageStatsResponse` | ✅ 응답 작음 |
| 집계 시점 | **히트맵만 import 때 사전계산.** 나머지는 요청마다 재계산 | `ActivityServiceImpl.monthlyCounts`(매번 `stream().collect`) | ⚠️ 요청당 O(n) |
| Explorer `/file` | 원본 JSON을 String으로 응답에 임베드, 10MB cap | `ExplorerFileResponse.content`, `ExplorerServiceImpl`(maxBytes) | ⚠️ 유일하게 무거운 전송 |
| 미지 파일 처리 | 미매칭 파일 → MISC_LOG catch-all로 격하 | `DefaultFileScanner` + `MiscLogParser` | ✅ 토대 있음 |
| 파일→도메인 매핑 | YAML 외부화(재컴파일 없이 규칙 추가) | `ScanMappingProperties` / `instagram-schema-mapping.yaml` | ✅ 토대 있음 |
| 스택 | Spring Boot 4.0.6 / Java 21 / Jackson 3. **DataSource·JPA 흔적 0(그린필드)** | `build.gradle.kts` | ✅ DB 도입에 유리 |

### 2.1 왜 "전송"이 병목이 아닌가
1. **localhost 단일 사용자**(설계 원칙: portable·single-user). 브라우저↔서버가 같은 머신 → 대역폭 공짜.
2. **대시보드 응답이 이미 집계값.** 게시물이 1억 개여도 `/api/activity`는 "월별 카운트 몇십 줄"만 내려간다. 데이터가 100배여도 응답 크기는 거의 그대로.

→ 차트/대시보드 전송은 데이터가 100배 늘어도 문제없다.

### 2.2 진짜 벽: 메모리
`InMemoryImportStore`는 `List<Post>`·`List<LikeEntry>`·`List<Conversation>` … 를 전부 힙에 보관.

- 지금: 수백 MB 수준, 견딤.
- 100배(수천만): 항목당 100~300B 잡아도 **수십 GB 힙 → 노트북 OOM 확정.**
- 억 단위: 인메모리 객체 방식으로는 물리적으로 불가능.
- 게다가 클릭마다 그 리스트를 `stream().collect`로 다시 훑음(요청당 O(n) CPU).

## 3. 후보 비교

선택 제약: **임베디드(서버 금지) + Java 접근 + 노트북에서 대용량 집계 + portable 단일 배포.**

| 후보 | 형태 | 대용량 집계 | 패키징 | 트레이드오프 |
|---|---|---|---|---|
| **DuckDB** ⭐ | 임베디드 **컬럼형 OLAP** | ⭐⭐⭐⭐⭐ (1억 행 GROUP BY ms급) | 네이티브 lib(JNI), OS별 바이너리 동봉 | 분석 최강. 단 순수 Java 아님, 잦은 소량 쓰기 부적합, Spring Data 공식 dialect 없음(→ JdbcClient/jOOQ) |
| SQLite | 임베디드 행기반 OLTP | ⭐⭐⭐ (수백만까진 OK, 1억 집계는 느림) | 작은 네이티브 lib | 최고로 검증·경량. 행 저장이라 풀스캔 집계 약함 |
| H2 | **순수 Java** 임베디드 관계형 | ⭐⭐ (중간까지) | **네이티브 없음**(가장 단순) | 추가 바이너리 0, 통합 쉬움. 대용량 분석 부적합 |
| Parquet + DuckDB/Arrow | 컬럼형 **파일 포맷** | ⭐⭐⭐⭐⭐ | DB 아님 | 초대용량 아카이브엔 이상적이나 구조 복잡, 현 단계 오버킬 |
| 인메모리 유지(원시배열/컬럼화) | 현행 개선 | ⭐⭐⭐ | 변경 최소 | 결국 RAM 한계 → "억 단위" 벽 못 넘음. 근본 해결 아님 |
| DuckDB-WASM(브라우저) | 프론트 직접 | ⭐⭐⭐⭐ | 백엔드 불필요 | 데이터가 서버로 안 감(프라이버시 최상, 철학 부합). 단 브라우저 메모리 한계·WASM 큼 |

### 3.1 NoSQL은?
"스키마 없이 아무 JSON이나 저장" 때문에 문제 B 측면에서 끌리는 건 합리적. 그러나:

| 유형 | 예시 | 문제 |
|---|---|---|
| 서버형 문서/검색 | MongoDB, Elasticsearch | **별도 서버 프로세스** → portable·무설치·더블클릭 실행 모델을 깸. 포트 충돌(이미 8374 이슈 경험), 데몬 RAM, OS별 바이너리 동봉. 게다가 OLAP 집계가 컬럼형 SQL보다 느림 |
| 임베디드 문서/KV | Nitrite, ObjectBox, RocksDB | 서버 문제는 없지만 **컬럼형 아님 → 빠른 GROUP BY 없음 → Java 손집계 = 요청당 O(n) 회귀.** 생태계도 작음 |

→ **"SQL이냐 NoSQL이냐"는 여기선 가짜 양자택일.** DuckDB/SQLite는 **JSON 컬럼**을 지원하고 DuckDB `read_json_auto`는 임의 JSON 스키마를 자동 추론한다. 즉 **문서형의 유연함 + SQL의 분석 속도**를 하이브리드로 얻는다. NoSQL은 "너무 비싸다"기보다 **비용이 엉뚱한 차원(portable 제약 파괴)에 들고, 끌렸던 유연함은 DuckDB로 공짜로 얻는다.**

## 4. 권장 해법

### 4.1 문제 A(양 불확실) → 예측이 필요 없게 설계
원리: **"용량을 예측"이 아니라 "비용이 데이터에 비례 + 메모리 상한".**

- **임포트**: 이미 스트리밍 → 피크 메모리가 파일 크기와 무관. 인플루언서 데이터도 "오래 걸릴 뿐" 안 터짐.
- **저장/조회**: DuckDB에 `memory_limit` 설정 → 초과분 디스크 spill → **어떤 규모든 OOM 없음.** 라이트 유저=작은 db 파일, 인플루언서=큰 db 파일, 쿼리 속도는 둘 다 빠름(컬럼형).

→ "양을 모른다"는 불확실성이 사라진다. 인메모리는 양을 알아야 안전, 디스크 기반은 양을 몰라도 안전.

### 4.2 문제 B(스키마 불확실) → 열린 집합으로 설계 (2-tier)
핵심 사실: 인스타 export는 무한 다양이 아니라 **거의 모든 파일이 ~4형태의 변주**
(`string_map_data` / `label_values` / `string_list_data` / `media_list_data` + `timestamp`).
→ 하나의 범용 평면화기로 대부분의 미지 파일 커버 가능(`MiscLogParser`가 이미 이용 중).

```
Tier 1 (큐레이션) : 아는 13개 도메인 → 타입드 테이블 + 전용 대시보드
Tier 2 (제네릭)   : 그 외 전부 → (file, root_key, timestamp, key, value) 범용 테이블
                    → 전용 UI 없어도 타임라인/검색 가능, 새 파일 자동 등장
                    → DuckDB read_json_auto 로 적재 비용 거의 공짜
폴백             : 그래도 안 되면 Explorer raw 로 원본 그대로 열람
```

**불확실성을 "보이게" 만들기** — 불안의 진짜 해법:
- **임포트 리포트**: "총 N개 파일 → 매칭(도메인별)/미매칭→제네릭/경고 Z건"을 사용자에게 표시. 조용한 위험 → 가시적 목록.
- (선택) **익명 스키마 리포트 내보내기**: 값 제외, "파일명+키 구조"만 추출 → 사용자가 자발적으로 공유 → 파서 개선. 데이터는 로컬에만.
- **테스트 코퍼스 구축**: 다른 로케일/비즈니스 계정 export로 파서 검증.

### 4.3 보조 개선 (DB 도입과 별개로도 유효)
- 집계 사전계산을 전 도메인으로 확장(히트맵 패턴) → 요청당 O(n) 제거. ※ DuckDB 가면 빠른 GROUP BY로 대체.
- **행 단위 조회 엔드포인트에만** 페이지네이션(`limit/offset` 또는 keyset). 집계 엔드포인트는 그대로.
- Explorer `/file`: 10MB String 임베드 → raw 스트리밍 엔드포인트로.
- 프론트: 행 리스트 뷰만 가상 스크롤(react-window). 차트는 집계라 무관.
- 런처 힙 가드: `-XX:MaxRAMPercentage=75` (현재 `-Xmx` 없음). DB 가면 거의 불필요.

## 5. Spring Boot ↔ DuckDB 연동성

**한 줄: JDBC 레벨에서 매끄럽고, 이 프로젝트엔 특히 잘 맞는다.**

| 항목 | 상태 | 비고 |
|---|---|---|
| DataSource / 커넥션 | ✅ | `jdbc:duckdb:/경로/analyze.duckdb` |
| **JdbcClient**(Spring 6.1+, Boot 4 기본) | ✅ **권장** | 분석 SQL에 딱 |
| JdbcTemplate / jOOQ / MyBatis | ✅ | 표준 JDBC라 다 됨 |
| `@Transactional` | ✅ | DuckDB ACID, DataSourceTransactionManager 동작 |
| 네이티브 라이브러리 | ⚠️→무관 | 드라이버 jar에 OS별 바이너리 동봉·런타임 추출. jpackage로 OS별 빌드라 문제없음 |
| **JPA / Hibernate** | ❌ | 공식 dialect 없음. **쓰지 말 것**(OLAP엔 ORM 부적합) |
| Spring Data JPA 리포지토리 | ❌ | JdbcClient로 직접 SQL |
| Flyway / Liquibase | ❌ | 임포트 때 `CREATE TABLE` 모델이라 마이그레이션 도구 사실상 불필요 |
| 커넥션 풀(Hikari) | ⚠️ | 됨. 단 단일-writer·읽기-다중이라 풀 작게(또는 롱리브드 1개) |

**왜 이 프로젝트에 특히 잘 맞나**
1. **기존 JPA 없음** → ORM과 싸울 일 없음(대부분의 "Spring+DuckDB 불편"은 JPA로 붙이려다 생김).
2. **워크로드가 SQL 분석** → `ActivityServiceImpl`의 Java 스트림 집계가 SQL 한 줄로.
3. **임포트 때 테이블 생성 모델** → 마이그레이션 도구 부재 무해.
4. **단일 사용자** → 동시성/풀링 고민 거의 없음(단일-writer 제약 무해).

붙이는 모습(개념 예시):
```kotlin
// build.gradle.kts
implementation("org.duckdb:duckdb_jdbc:<버전>")
```
```java
@Bean
DataSource duckDb(@Value("${app.data-dir}") String dir) {
    var ds = new DriverManagerDataSource("jdbc:duckdb:" + dir + "/analyze.duckdb");
    ds.setDriverClassName("org.duckdb.DuckDBDriver");
    return ds;
}
```
```java
// ActivityServiceImpl 의 stream().collect 자리
List<MonthlyCount> rows = jdbcClient.sql("""
    SELECT strftime(to_timestamp(ts_millis/1000), '%Y-%m') AS month, COUNT(*) AS cnt
    FROM activity WHERE type = ? GROUP BY month ORDER BY month
""").param(type.name()).query(MonthlyCount.class).list();
```

`ImportReadStore` / `ImportWritePort` 포트 추상화가 이미 있어 **구현체만 DuckDB 버전으로 교체**하면 컨트롤러·서비스 상당부는 그대로 둘 수 있다(설계가 잘 되어 있는 덕).

## 6. 의사결정 요약

| 걱정 | 진짜 성격 | 해법 |
|---|---|---|
| 양을 예측 불가 | 예측 자체가 불필요해야 함 | 스트리밍 임포트 + 디스크 기반 컬럼형(DuckDB) = 양과 무관 메모리 상한·속도 |
| 모르는 파일/스키마 | 무한 아님(~4형태 변주) | 2-tier(큐레이션+제네릭+raw) + 임포트 리포트 가시화 |
| 전송이 느릴까 | 병목 아님 | 집계 응답 + localhost. 행 조회만 페이징 |
| NoSQL 도입? | 비용이 엉뚱한 차원 | 서버형=portable 파괴 / 임베디드형=분석 약함. 유연함은 DuckDB JSON으로 |
| Spring 연동 | 양호 | JDBC + JdbcClient. JPA 금지. 그린필드+포트추상화로 깔끔 |

**최종 권고**: `InMemoryImportStore` → **DuckDB 기반 2-tier 스토어**로 전환. 이 한 결정이 "양 불확실 + 스키마 불확실 + 메모리 벽 + 요청당 O(n)"을 동시에 해소한다.
